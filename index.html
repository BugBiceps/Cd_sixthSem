Unit 1 — Notes

Lexical Analysis

Core concepts: what a compiler does, its phases, and how lexical analysis tokenises source text.

6
key topics to know
~20%
exam weight
★★☆
difficulty level
Compiler vs Interpreter
  • Compiler — translates entire source to target before execution. Faster at runtime, higher memory usage.
  • Interpreter — translates and executes line by line. Slower but easier to debug.
  • Assembler — converts assembly language to machine code (1-to-1 mapping).
Phases of a Compiler (must memorise)
  • 1. Lexical Analysis — reads source, produces tokens, removes whitespace/comments.
  • 2. Syntax Analysis — builds parse tree using grammar rules (CFG).
  • 3. Semantic Analysis — type checking, scope resolution, attribute evaluation.
  • 4. Intermediate Code Generation — produces 3AC / TAC / quadruples.
  • 5. Code Optimisation — improves intermediate code (speed/size).
  • 6. Code Generation — produces target machine code / assembly.
Lexical Analysis key terms
  • Token — <token-class, attribute> pair. e.g. <id, "x">, <op, "+">
  • Lexeme — actual character sequence in source that matches a pattern.
  • Pattern — rule (regex) describing a token class.
  • Input Buffering — two-buffer scheme to handle lookahead efficiently; each buffer ~4KB.
  • Sentinels — EOF marker placed at end of each buffer to avoid boundary checks.
LEX tool
  • LEX takes a .l file with regex rules and generates a C lexer.
  • Structure: definitions | rules (pattern → action) | user code.
  • Internally builds a DFA from the combined NFA of all patterns.
Exam tip: Always draw the 6-phase pipeline diagram and trace an expression like a := b + c * 4 through each phase — this appears almost every year.
Unit 2 — Notes

Syntax Analysis & SDT

The heaviest unit. Master CFG, First/Follow, parsing tables, and attribute grammars.

10
key topics to know
~35%
exam weight
★★★
difficulty level
Context-Free Grammar (CFG)
  • G = (V, T, P, S) — variables, terminals, productions, start symbol.
  • Ambiguous grammar — a string has more than one parse tree. Fix by adding precedence/associativity rules.
  • Left recursion — A → Aα | β. Eliminate: A → βA', A' → αA' | ε
  • Left factoring — A → αβ | αγ becomes A → αA', A' → β | γ
FIRST and FOLLOW
  • FIRST(α) — set of terminals that begin strings derivable from α. If α ⇒* ε, add ε.
  • FOLLOW(A) — terminals that can appear immediately after A in some sentential form. $ ∈ FOLLOW(S).
  • Used to build LL(1) and SLR parsing tables.
Top-Down vs Bottom-Up Parsing
  • Top-Down — starts from start symbol, expands toward input. LL(1), recursive descent.
  • Bottom-Up — starts from input, reduces to start symbol. LR(0), SLR(1), LALR(1).
  • LL(1) — Left-to-right scan, Leftmost derivation, 1 token lookahead. No left recursion allowed.
SLR / LALR Table Construction
  • Step 1 — Augment grammar: add S' → S.
  • Step 2 — Compute LR(0) canonical items (closure + goto).
  • Step 3 — Build ACTION table: shift on terminal, reduce when dot at end (using FOLLOW for SLR).
  • Step 4 — Build GOTO table: for non-terminals in each state.
  • LALR — merges states with same core items; smaller table than LR(1), same power as SLR for most grammars.
Syntax Directed Translation (SDT)
  • Synthesized attribute — value computed from children. Used in S-attributed SDTs.
  • Inherited attribute — value passed from parent/sibling. Used in L-attributed SDTs.
  • S-attributed — only synthesized attributes. Safe to evaluate during bottom-up parse.
  • L-attributed — inherited attrs depend only on left siblings or parent. Safe with top-down parse.
  • Annotated parse tree — parse tree with attribute values at each node.
Exam tip: Practice SLR/LALR table construction from scratch — it's a guaranteed numerical. Draw the LR(0) item sets step by step.
Unit 3 — Notes

Type Checking & Runtime Environment

How the compiler enforces type rules and how memory is organised at runtime.

5
key topics to know
~12%
exam weight
★★☆
difficulty level
Type Checking
  • Type checker — semantic analysis phase that verifies type consistency of operations.
  • Static type checking — at compile time (C, Java). Dynamic — at runtime (Python).
  • Type equivalence — Structural (same shape) vs Name equivalence (same declaration).
  • Polymorphism — function works on multiple types. Ad-hoc (overloading), parametric (generics), subtype.
Storage Allocation Strategies (high priority)
  • Static allocation — memory fixed at compile time. No recursion. e.g. global vars, FORTRAN.
  • Stack allocation — activation records pushed/popped on call/return. Supports recursion. LIFO.
  • Heap allocation — dynamic memory via malloc/new. Flexible but needs GC or manual free.
Activation Record (AR)
  • Contains: return value, actual parameters, control link (caller's AR), access link, saved machine state, local variables, temporaries.
  • Control link — points to caller's AR (dynamic chain).
  • Access link — points to AR of statically enclosing scope (static chain).
Exam tip: Draw the activation record layout and explain stack vs heap with a diagram. Storage allocation appears in almost every paper it shows up in.
Unit 4 — Notes

Code Generation

Intermediate representations, basic blocks, flow graphs, and DAGs — all practical, all numerical.

8
key topics to know
~25%
exam weight
★★★
difficulty level
Three-Address Code (3AC)
  • At most one operator per instruction. Form: x = y op z
  • Quadruples — (op, arg1, arg2, result). Easy to reorder; preferred in optimising compilers.
  • Triples — (op, arg1, arg2); result implicit as index. Compact but hard to reorder.
  • Indirect triples — list of pointers to triples; reordering easier than triples.
  • Back patching — filling in incomplete jump targets once the destination is known.
Basic Blocks & Flow Graphs
  • Basic block — maximal sequence of instructions with one entry, one exit, no internal jumps.
  • Leaders — first instruction, any branch target, instruction after a branch.
  • Flow graph — directed graph where nodes = basic blocks, edges = control flow.
DAG (Directed Acyclic Graph)
  • Represents computations in a basic block. Leaves = inputs; interior nodes = operators.
  • Detects common sub-expressions automatically (shared nodes).
  • Algorithm: for each instruction x=y op z, find/create nodes for y,z; create node for op; label with x.
  • Applications: CSE elimination, dead code removal, algebraic simplification.
Register Allocation & Peephole
  • Register allocation — assign variables to registers; use graph colouring (interference graph).
  • Peephole optimisation — small sliding window over code; eliminates redundant loads/stores, unreachable code, jumps to jumps.
Exam tip: Practice DAG construction and 3AC generation by hand. Draw the flow graph for while/for loops step by step.
Unit 5 — Notes

Code Optimization

Techniques to improve generated code without changing its meaning — mostly theoretical, high repetition in exams.

6
key topics to know
~12%
exam weight
★★☆
difficulty level
Sources of Optimization (most repeated topic)
  • Common Sub-expression Elimination (CSE) — avoid recomputing the same expression.
  • Copy Propagation — replace copies (x=y) with the original variable y.
  • Dead Code Elimination — remove code whose result is never used.
  • Constant Folding — evaluate constant expressions at compile time. e.g. 3+4 → 7
  • Strength Reduction — replace expensive op with cheaper one. e.g. x*2 → x+x or x<<1
Loop Optimisations
  • Loop-invariant code motion — move computations that don't change across iterations outside the loop.
  • Induction variable elimination — simplify variables that change by a fixed amount each iteration.
  • Loop unrolling — replicate loop body to reduce branch overhead.
Three Areas of Optimisation
  • Local — within a single basic block. e.g. peephole, CSE in block.
  • Global — across basic blocks in a function. Uses data flow analysis.
  • Inter-procedural — across function boundaries. Most expensive.
Global Data Flow Analysis
  • Computes properties of the program at each point using data flow equations.
  • Reaching definitions — which assignments can reach a given point.
  • Live variable analysis — which variables will be used after this point.
  • Solved iteratively (forward or backward) until a fixed point is reached.
Exam tip: Learn the names and one-line definitions of all sources of optimisation — they appear as "write a short note on" questions almost every year.
